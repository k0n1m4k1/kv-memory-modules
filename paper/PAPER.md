# Precompiled KV Memory Modules: Relocatable, Composable Agent Memory for LLM Inference Runtimes

*Luis Andrés González Picazo*
*v1.0 — July 21, 2026*

---

## Abstract

Persistent memories for LLM agents are stored today as text files (Markdown) that are re-prefilled every session, or indexed in RAG systems that re-inject text fragments. The same holds for the skill, plan, and agent-definition files (`SKILLS.md`, `PLANS.md`, `AGENTS.md`) and the JSON tool schemas an agent loads, whether directly or over MCP: all of it is re-evaluated by the LLM on every new session and access — compute time that consumes resources on every cold start. We propose treating each memory as a **precompiled KV module**: the attention state (per-layer K/V tensors) produced by evaluating the text once, serialized to disk with content-addressed identity, and **re-insertable at any position** of a live context via RoPE position re-rotation and sequence fusion — without re-evaluating the text. Using stock llama.cpp release binaries, across nine instruction-tuned models (2B–14B; four full-attention, four linear-attention hybrids — one carrying an MTP draft head — and one interleaved sliding-window model) and memories up to 51.8k tokens, we demonstrate: (1) **non-prefix insertion with no statistically detectable recall difference** versus joint prefill (paired McNemar exact *p*=0.69; 95 % CI on the deficit [−1.7, +3.1] pp over 420 core paired questions, §5.2), under adversarial prefixes, replicated bit-for-bit in recall across OS, GPU vendor, and backend (Windows/Vulkan/Arc vs. Linux/CUDA/RTX); (2) a setup-cost advantage that is a function of available compute — restoring a 15.2k-token module from disk takes a flat ~0.7 s while prefilling it takes 5.5 s on a discrete GPU, 13.7 s under partial layer offload, and 18.9 s on CPU (**7.0–27.6×**, median of five runs) — a ratio that also grows with model size (3.5× at 14B) and memory length (6.1× at 51.8k tokens) on the same GPU, and barely moves when the module is read cold from NVMe rather than the OS page cache (0.78 s vs 0.65 s at full GPU: state relocation dominates, not the disk) — which makes precompilation most valuable precisely on consumer and edge hardware; (3) two-module composition with a characterized, statistically confirmed 10–20 % attribution deficit (pooled McNemar *p*<0.001), **reduced by recomputing ~⅓ of the inserted module** (splice-k) and closing to parity at scale — a 33.4k-token three-module workspace matches joint prefill (106 vs. 105 of 120; McNemar *p*=1.0); (4) lazy loading of `[[linked]]` modules mid-conversation (0/6 → 6/6 after a 1.1 s link of a 10.2k-token module); (5) **extension of the linker to linear-attention hybrids** (Qwen3.5 / Gated DeltaNet): recurrent state links as a constant-size affine pair `(T_M, S_M)` extracted via per-layer probes with no runtime modification, reaching full recall parity with joint prefill at 2B, 4B and 9B; (6) a module ABI: KV-cache dtype is model- and length-dependent (q8_0 matches f16; q4/q5 collapse silently on some models past ~9k tokens), backend is *not* a compatibility axis, modules are machine-portable artifacts, and interleaved sliding-window attention links unmodified — the module inherits the window's own visibility semantics (§5.9); and (7) **compatibility with speculative decoding via multi-token-prediction (MTP) draft heads**: we identify the serialization gap that silently drops draft-head KV in llama.cpp, close it with a ~120-line library patch, and show that restored state preserves draft acceptance (0.72 vs. 0.69 full-prefill baseline; removing only the draft blob degrades it to 0.59 with a 9 % throughput loss while answers stay correct). These results identify the "memory linker" — absent today from both llama.cpp and vLLM — as the missing piece for agent memory managers with near-zero warm-up cost. The linker is implemented and evaluated on llama.cpp; the vLLM result (E9) is a prefix-restore feasibility check and the cross-runtime `.kmd` format a proposal (§6.4), not yet a demonstrated vLLM linker. To our knowledge, this is the first system to give KV state an *artifact* lifecycle — persistent, portable, content-addressed, staleness-checked — rather than a cache-entry lifetime, the first KV linker of any kind for hybrid linear-attention models, and the first serialization of MTP draft-head KV state in a consumer runtime; everything runs on unmodified release binaries, with no engine fork, custom kernels, or fine-tuning (the sole exception is the §5.8 MTP result, which requires our upstream-candidate serialization fix — vLLM needs none, §6.4).

## Resumen (Español)

Las memorias persistentes de los agentes LLM se almacenan hoy como archivos de texto (Markdown) que se reprocesan (*prefill*) en cada sesión, o se indexan en sistemas RAG que reinyectan fragmentos de texto. Lo mismo ocurre con `SKILLS.md`, `PLANS.md`, `AGENTS.md` y la definición en JSON de las herramientas, sea de forma directa o por MCP: todo ello es reevaluado por el LLM en cada nueva sesión y acceso — tiempo de cómputo que consume recursos en cada arranque en frío. Proponemos tratar cada memoria como un **módulo KV precompilado**: el estado de atención (tensores K/V por capa) resultante de evaluar el texto una sola vez, serializado a disco con identidad direccionada por contenido, y **re-insertable en cualquier posición** de un contexto vivo mediante re-rotación RoPE de posiciones y fusión de secuencias — sin re-evaluar el texto. Sobre binarios oficiales de llama.cpp, con nueve modelos (2B–14B; cuatro de atención completa, cuatro híbridos de atención lineal —uno con cabezal draft MTP— y uno de ventana deslizante intercalada) y memorias de hasta 51,8k tokens, demostramos: (1) **inserción no-prefijo sin diferencia de recall estadísticamente detectable** frente al prefill conjunto (McNemar pareado exacto *p*=0,69; IC 95 % del déficit [−1,7, +3,1] pp, §5.2), con prefijos adversarios, replicada con puntuaciones idénticas entre SO, fabricante de GPU y backend (Windows/Vulkan/Arc vs. Linux/CUDA/RTX); (2) una ventaja de coste de arranque que es función del cómputo disponible — restaurar un módulo de 15,2k tokens desde disco cuesta ~0,7 s planos mientras su prefill cuesta 5,5 s en GPU dedicada, 13,7 s con offload parcial de capas y 18,9 s en CPU (**7,0–27,6×**, mediana de cinco ejecuciones) — una razón que además crece con el tamaño del modelo (×3,5 a 14B) y con la longitud de la memoria (×6,1 a 51,8k tokens) en la misma GPU, y apenas se mueve aunque el módulo se lea en frío desde el NVMe en lugar de la page cache del SO (0,78 s vs 0,65 s a plena GPU: domina la relocación del estado, no el disco) — lo que hace la precompilación más valiosa justo en hardware doméstico y de borde; (3) composición de módulos con un déficit de atribución del 10-20 % confirmado estadísticamente (McNemar *p*<0,001), **reducido recomputando ~⅓ del módulo insertado** (splice-k) y con paridad recuperada a escala — un workspace de 33,4k tokens y tres módulos iguala al prefill conjunto (106 vs. 105 de 120; McNemar *p*=1,0); (4) carga perezosa de módulos `[[enlazados]]` en mitad de la conversación (0/6 → 6/6 tras un enlace de 1,1 s de un módulo de 10,2k tokens); (5) **extensión del linker a híbridos de atención lineal** (Qwen3.5 / Gated DeltaNet): el estado recurrente se enlaza como par afín `(T_M, S_M)` de tamaño constante extraído con sondas por capa sin modificar el runtime, con paridad total de recall en 2B, 4B y 9B; (6) un ABI de módulos: el dtype de la cache KV depende del modelo y la longitud (q8_0 iguala a f16; q4/q5 colapsan silenciosamente en algunos modelos a partir de ~9k tokens), el backend *no* es eje de compatibilidad, los módulos son artefactos portables entre máquinas, y la atención de ventana deslizante intercalada enlaza sin cambios heredando la visibilidad de la propia ventana (§5.9); y (7) **compatibilidad con la decodificación especulativa vía cabezales de predicción multi-token (MTP)**: identificamos el hueco de serialización que descarta silenciosamente el KV del cabezal draft en llama.cpp, lo cerramos con un parche de ~120 líneas en la librería, y mostramos que el estado restaurado preserva la aceptación especulativa (0,72 vs. 0,69 de baseline con prefill completo; retirar solo el blob draft la degrada a 0,59 con −9 % de throughput, con respuestas correctas). Los resultados sitúan el "linker de memorias" —inexistente hoy en llama.cpp y vLLM— como la pieza que falta para un gestor de memorias de agente con coste de arranque casi nulo. El linker está implementado y evaluado sobre llama.cpp; el resultado en vLLM (E9) es una comprobación de viabilidad de restauración prefijo y el formato `.kmd` entre runtimes una propuesta (§6.4), no todavía un linker demostrado en vLLM. Hasta donde sabemos, este es el primer sistema que da al estado KV un ciclo de vida de *artefacto* —persistente, portable, direccionado por contenido, con verificación de obsolescencia— en lugar de una vida de entrada de caché, el primer linker KV de cualquier tipo para modelos híbridos de atención lineal, y la primera serialización del estado KV de cabezales MTP en un runtime de consumo; todo sobre binarios oficiales sin modificar, sin forks del motor, kernels a medida ni fine-tuning (la única excepción es el resultado MTP de §5.8, que requiere nuestra corrección de serialización candidata a upstream — vLLM no necesita ninguna, §6.4).

## 1. Introduction

Agent frameworks persist long-lived knowledge — user preferences, project state, conventions — as natural-language files (typically Markdown) injected into the context at session start; the same is true of the skill, plan, and agent-definition files (`SKILLS.md`, `PLANS.md`, `AGENTS.md`) and the JSON tool schemas an agent loads, directly or over MCP. This design pays the *prefill tax* on every session: the same tokens are re-evaluated through the full transformer stack, at a cost that grows with memory size and model depth, on every cold start. Retrieval-augmented approaches reduce *what* is injected but still re-evaluate whatever is retrieved.

The key observation is an analogy with managed runtimes: a Markdown memory is *source code*; the KV-cache entries produced by prefilling it are its *compiled form*; and the inference runtime is a *virtual machine* that can, in principle, load compiled artifacts directly. Java does not re-run `javac` on every JVM start. This work asks: **can agent memories be compiled once into KV artifacts and linked into live contexts at arbitrary positions, across sessions, processes, and backends?**

Prefix caching (in llama.cpp, vLLM, and commercial APIs) already answers a restricted form of this question: compiled prefixes can be reused *if and only if the new context starts with the exact same tokens*. This restriction is fatal for agent memory: any variable content before the memory — a date, a greeting, a conversation turn — invalidates the entire cached artifact (we quantify this in §5.1). What is missing is the **linker**: relocation of a compiled module to a new position, and composition of independently compiled modules. Concurrent serving-side work addresses relocation for datacenter engines (position-independent caching, §2); no linker ships in the runtimes themselves, and none covers persistent memory artifacts, consumer hardware, or hybrid architectures.

Contributions:

1. A code-level analysis of KV persistence in llama.cpp and vLLM, identifying the existing primitives (serialization, RoPE K-shift relocation, sequence fusion) and the structural prefix-only constraint in both (§3).
2. A working memory linker for llama.cpp built entirely on public APIs of unmodified release binaries (§4).
3. An experimental evaluation on four full-attention models (Qwen3-4B, Llama-3.2-3B, Qwen2.5-Coder-7B, Qwen3-14B) and memories from 1.4k to 51.8k tokens showing single-module insertion with no statistically detectable recall difference from joint prefill (§5.2), a characterized multi-module attribution deficit that shrinks with model capability and is reduced by splice-k recomputation (fully closed to parity at workspace scale, not at the two-module micro-scale), a 33.4k-token three-module workspace at parity with joint prefill, and a viable lazy-loading protocol for linked modules (§5.2–5.4, §5.7).
4. An extension of the linker to linear-attention hybrids (Qwen3.5 at 2B/4B/9B, Gated DeltaNet): recurrent state linked as a constant-size affine pair extracted via per-layer probes, plus a host-side software RoPE rebase that removes the runtime's M-RoPE relocation restriction (§5.5).
5. A cross-hardware, cross-runtime replication (Windows/Vulkan/Arc ↔ Linux/CUDA/RTX; llama.cpp ↔ vLLM) and a compute-regime cost model showing that the linker's O(bytes) advantage over prefill's O(compute) grows exactly where compute is scarce — consumer CPUs and partial layer offload — and additionally with model size and memory length, and does not depend on the module being cached in RAM: a cold NVMe read costs essentially the same as a warm one (~0.1 s more, §5.6).
6. A module ABI analysis (what breaks compatibility, including a model- and length-dependent KV-quantization cliff) and a runtime-agnostic interchange format proposal with content-addressed identity, implemented as the `.kmd` container and `mdc` CLI (§6).
7. Compatibility with MTP speculative decoding: a code-level identification of the draft-head serialization gap in llama.cpp (shared-cell KV state is silently dropped), a ~120-line upstream-candidate fix, and a causal validation on a hybrid+MTP model — restored state preserves draft acceptance, and ablating only the draft blob degrades it (§5.8).
8. A measured **context-virtual-memory layer** built from the same primitives: in-conversation eviction with compaction (0.5–1 ms full-attention; ~5 ms checkpoint-and-replay for hybrids), sealing live conversation segments as relocatable modules (a 5.5k-token conversation lives in a 4k window with 142 ms page-ins), and paged reading of a 51.8k-token document under a 4k budget that *outscores* full-context prefill by 15 points (§6.6–§6.7).

Relative to the concurrent position-independent-caching line (§2), four properties of this work are, to our knowledge, unique. First, **modules are artifacts, not cache entries**: compiled once, stored on disk with content-addressed identity, checked for staleness, and reused across sessions, processes, machines, and backends — all measured, not assumed (§5.6). Second, **the mechanism requires nothing from anyone**: unmodified release binaries of a consumer runtime, no engine fork, no custom kernels, no fine-tuning — every ingredient already ships. Third, **architecture coverage**: this is the first KV linker for linear-attention hybrids, where a module becomes a constant-size affine operator on recurrent state (§5.5) — arguably a more natural compiled memory than per-token KV. Fourth, **the economics are characterized, not just the mechanism**: the advantage is a function of compute scarcity (7.0–27.6× for the same module, §5.6), peaking exactly on the consumer and edge hardware that serving-side systems do not target. Beyond mechanism, the evaluation contributes an empirical *bound* on the phenomenon that approximation-based systems model: prefix-induced KV deviation is behaviorally negligible for single modules up to 15k tokens (no statistically detectable difference over 420 paired questions, §5.2), and where it does matter — multi-module attribution — a small exact recomputation reduces it, closing to parity at workspace scale (§5.3, §5.7).

## 2. Related Work

**KV state serialization and prefix reuse.** llama.cpp exposes per-sequence state save/restore (`llama_state_seq_*`) and server-level slot persistence; its RAM prompt cache reuses states by longest common prefix. vLLM (Kwon et al., 2023) implements Automatic Prefix Caching by addressing KV blocks with chained content hashes; SGLang's RadixAttention (Zheng et al., 2024) organizes reusable prefixes in a radix tree. vLLM's `KVConnectorBase_V1` interface supports external KV stores — LMCache (LMCache Team, 2024), Mooncake (Qin et al., 2025), NIXL — with asynchronous, layer-wise injection into paged memory. All of these reuse state only at identical prefix positions.

**Modular and non-prefix reuse.** Prompt Cache (Gim et al., 2024) proposed modular attention-state reuse with a schema of interchangeable segments. CacheBlend (Yao et al., 2025) fuses independently compiled KV chunks at non-prefix positions by selectively recomputing a small fraction of tokens, recovering most of the quality gap. CacheGen (Liu et al., 2024) compresses KV states for storage and transmission.

**Position-independent caching (PIC).** Non-prefix KV reuse has since crystallized into a named research area on the serving side: EPIC (Hu et al., 2025) formalizes PIC and mitigates the per-chunk attention-sink effect with static sparse recomputation; KVLink (J. Yang et al., 2025) adjusts positional embeddings at concatenation time and fine-tunes the model with trainable link tokens; KVCOMM (Ye et al., 2025) *estimates* prefix-induced cache deviation from an online pool of anchor examples; MiniPIC (Ordonez and Parnell, 2026) hosts PIC primitives in a vLLM fork with under 100 lines of engine changes; and KEEP (Z. Yang et al., 2026), closest in motivation, manages embodied-agent memory as KV cache groups inside one vLLM instance (~2.8k LOC extension) — an online cache manager whose entries are neither persisted, relocated, versioned, nor portable, for full-attention models only. All of this line lives inside modified serving engines and targets datacenter deployment; the four properties unique to this work are listed at the end of §1, and the capability-by-capability gap is mapped below.

**Virtual context management.** MemGPT (Packer et al., 2023) introduced the operating-system metaphor for agent memory: a bounded main context backed by external storage, with the model paging information in and out via function calls. That metaphor operates at the *text* level — paged-out content is summarized (lossily) or stored as text, and page-in re-prefills it, paying exactly the tax this work eliminates. vLLM's PagedAttention (Kwon et al., 2023) applies the same metaphor one level down, to the *physical* layout of KV blocks in GPU memory, without extending what a sequence can address; engine-internal KV tiering (InfLLM-style memory units, KEEP's GPU/CPU groups) pages attention state but inside one modified serving process, with no persistence, relocation, or portability. Our modules supply the missing middle layer: page-in/page-out of *compiled state* on stock runtimes, at exact-state fidelity and O(bytes) cost, with measured in-conversation eviction (§6.6) — which turns MemGPT-style virtual context from a summarize-and-reprefill protocol into an actual page file for attention state. What gets extended is the *addressable* context (bounded by storage); the resident attention window remains bounded by the trained context length.

**Attention sinks and hybrid architectures.** StreamingLLM (Xiao et al., 2024) showed that initial-position tokens act as attention sinks; module insertion changes what occupies the early positions a module "grew up with", and our sink-dropping ablations (§5.3) probe this directly. Gated DeltaNet (S. Yang et al., 2025) is the linear-attention recurrence used by the Qwen3.5 hybrids to which §5.5 extends the linker.

**What exists, and what was missing.** We are explicit about what this work does not invent: state serialization, RoPE relocation, sequence fusion, non-prefix insertion, and the virtual-memory metaphor all pre-exist, and this section credits each. The claim is the vacant cell, per capability:

| Capability | Closest prior art | Its scope | Missing until this work |
|---|---|---|---|
| Non-prefix KV insertion | CacheBlend, EPIC, KVLink | inside modified/fine-tuned serving engines | on unmodified consumer release binaries, from native primitives (§4) |
| KV persistence across requests | LMCache, Mooncake; llama.cpp session files | prefix-keyed cache entries inside one serving stack | content-addressed, staleness-checked *artifacts*, relocatable and portable across machines and backends (§5.6, §6.1) |
| Virtual context / paging | MemGPT (text), PagedAttention (physical blocks) | summarize-and-reprefill; GPU block layout | paging of *compiled state* at exact-state fidelity, with ms-scale measured eviction (§6.6) |
| Modular prompt reuse | Prompt Cache | schema-fixed positions, custom implementation | schema-free relocation to arbitrary positions, no custom kernels (§4) |
| Hybrid (linear-attention) state reuse | vLLM Mamba/GDN prefix snapshots | prefix-shaped, engine-internal | non-prefix linking as a constant-size affine operator — the first KV linker for hybrids (§5.5) |
| MTP draft-head KV | LMCache DeepSeek-MTP layers | prefix-shaped, datacenter stack | first serialization in a consumer runtime, causally validated, with an upstream-candidate fix (§5.8) |
| In-conversation eviction | KEEP (engine-internal importance groups) | online cache manager, no persistence or relocation | measured evict+compact on live conversations, attention and recurrent paths alike (§6.6) |

Each mechanism row pre-exists in some form; no prior system fills any entry of the last column, and no combination of them yields the artifact lifecycle that persistent agent memory needs. That combination — not any single trick — is the contribution.

## 3. Background: what the runtimes already provide

**llama.cpp.** A sequence's state is serialized as: prompt tokens, then per KV cell its absolute position and sequence ids, then per layer the raw K rows and V rows (`llama-kv-cache.cpp:1957-2200`; file container at `llama-context.cpp:3105`; all code references at tag b10068). On restore, *physical* cells are re-allocated freely, but *logical* positions are restored verbatim — no relocation is applied at load. Separately, `llama_memory_seq_add` marks per-cell position deltas and the next decode executes a K-shift graph that re-rotates K in place via RoPE, including for quantized caches (`llama-kv-cache.cpp:566, 1909`). V carries no positional information under RoPE and is untouched. Sequence membership can be copied cheaply between sequences in a unified KV buffer (`seq_cp`/`seq_rm`). **All linker ingredients exist; they are simply not connected.**

**vLLM.** Block identity is a chained hash of (parent hash, block tokens, extra keys) (`kv_cache_utils.py:596`), so a block's identity encodes its entire prefix: prefix-only reuse is enforced *structurally by the addressing scheme*, not incidentally. The connector API's scheduler contract (`get_num_new_matched_tokens`) asks how many tokens *beyond the computed prefix* the connector can supply — the contract itself is prefix-shaped. Non-prefix insertion therefore requires scheduler cooperation, not merely a new connector.

**Model-family constraints.** MTP contexts in llama.cpp share KV cells with the target context and all state operations silently no-op on them (`[TAG_KV_CACHE_SHARE_CELLS]`): MTP speculation is incompatible with state restore today. Classic two-model speculation is compatible (draft state is saved separately). M-RoPE (vision-language) models are excluded by `seq_add`'s single-position assertion; sliding-window attention and recurrent hybrids have partial-state semantics.

**Recurrent state is affine — so hybrid modules are constant-size.** The newest hybrid linear-attention generation (e.g., Qwen3.5, where 3 of every 4 layers use Gated DeltaNet recurrence, `src/models/qwen35.cpp`) is outside the *implemented* linker's reach — recurrent state has no per-token entries to relocate — but not outside its mathematical reach. Because these recurrences are linear in the state (`S_t = A_t S_{t-1} + B_t`), an entire module's effect is an affine operator: `S(P;M) = T_M · S(P) + S_M`, with `T_M = Π A_t` and `S_M` the module's zero-state response. A recurrent module is therefore a *constant-size* artifact (one `d×d` matrix plus one state per layer/head, regardless of module length), and linking is one matmul per layer — the same associativity that chunked-scan training algorithms already exploit, never yet externalized as a persistable artifact. Recurrent state is position-invariant, so no rebasing is even needed; the hybrid's full-attention layers link exactly as in §4.

Note that *prefix-shaped* persistence for these families is already production practice: vLLM snapshots Mamba/GDN states at block boundaries for prefix caching (`mamba_cache_mode: "all"`), and LMCache serializes MTP-layer KV for DeepSeek-MTP models — confirming that MTP and hybrid support are engineering gaps in llama.cpp, not mathematical ones. Sections 5.2–5.4 use standard-RoPE, full-attention decoder models; **§5.5 implements and validates the affine-composition linker on Qwen3.5 hybrids**, working around the M-RoPE `seq_add` exclusion with a host-side software rebase.

## 4. The memory linker

We implement compile, load, relocate, and fuse operations against unmodified llama.cpp b10068 release binaries, via Python `ctypes` bindings (~200 lines).

- **Compile** (`mdc` step): tokenize the memory text, prefill it alone as sequence 0 at positions `0..n-1`, and serialize with `llama_state_seq_save_file`. The artifact is the *module*.
- **Link** at position `P` of a live context: `llama_state_seq_load_file` into an auxiliary sequence; optionally drop leading cells; `llama_memory_seq_add(aux, -1, -1, P)` to rebase positions (the RoPE K-shift executes lazily on the next decode); `seq_cp(aux → main)` and `seq_rm(aux)` to fuse. Requires `kv_unified=true` and `n_seq_max ≥ 2`.
- **ABI pinning**: flash attention is pinned off because `v_trans = !flash_attn` changes the serialized V layout; the KV dtype is pinned to f16 for the §5.2–5.4 experiments (both axes are explored systematically in §6, including all nine cache dtypes and an FA↔non-FA converter).

The linker adds ~0.4–0.6 s per link for 31–297 MB modules on our hardware, dominated by disk I/O; the K-shift itself is a single fused graph pass.

- **Models.** Qwen3-4B-Instruct-2507 Q4_K_M (36 layers, GQA 8, f16 KV ≈ 147 KB/token), Llama-3.2-3B-Instruct Q4_K_M, Qwen2.5-Coder-7B-Instruct Q4_K_M (a scale and specialization point), and Qwen3-14B Q4_K_M (the largest full-attention point, §5.2); §5.5 adds the hybrids Qwen3.5-2B/4B Q4_K_M and Qwen3.5-9B Q8_0 (ChatML harness with the thinking mode pinned off — see the methodological note there); §5.8 adds Qwopus3.5-4B-Coder-MTP Q6_K, a GDN hybrid with an embedded MTP draft head; §5.9 adds Gemma-3-4B-it Q4_K_M (interleaved sliding-window attention). Full model properties and sources in Appendix A.
- **Decoding and scoring.** Greedy decoding throughout. Recall is scored by accent-insensitive substring match against facts that exist *only* in the memory text, verified by no-memory controls.
- **Corpora.** §5.2–5.4 use a synthetic 1.4k-token project memory and a 0.3k-token secondary memory; E8 uses a deterministic 5.1k-token generated memory (seeded generator in the script); E10–E12 use three Wikipedia-bulk memories (8.0k/10.2k/15.2k tokens) whose questions target *injected synthetic facts only* (fake expedient ids, assignees, budgets — absent from any pretraining corpus by construction; seeded generator `gen_corpus.py`), so parametric knowledge cannot inflate scores.
- **Isolation and hygiene.** Each question is asked from an identical base state, isolated by KV rollback (`seq_rm`) — we found and corrected a question-chaining confound that initially masqueraded as an insertion-quality deficit. Modules are recompiled per run and tagged per model; every link asserts the expected `seq_pos_max`.

### 5.1 Cold restore vs. prefill, and prefix fragility (Phase A)

Single module restored as a prefix through the llama.cpp server's slot-persistence API — the baseline the rest of the paper builds on.

| Metric | GPU (Vulkan) | CPU (same module file) |
|---|---|---|
| Cold restore (1380 cells, 194 MB) | 288 ms | 358 ms |
| Query after restore | 46 tok / 218 ms | 46 tok / 671 ms |
| Cold baseline (full prefill) | 1425 tok / 2196 ms | 1425 tok / 8668 ms |
| TTFT speedup | **4.3×** | **8.4×** |

Outputs are token-identical to the baseline. The same binary artifact restores across backends: serialization copies raw host-side bytes (`ggml_backend_tensor_get/set`), so the format is backend-agnostic (Vulkan/CUDA/SYCL/CPU); the empirical Vulkan→CPU transfer confirms it.

**Prefix fragility.** With a 47-token variable prefix (system + timestamp) prepended, the restored module contributes nothing: 1464/1464 tokens are reprocessed. One edited line in the source MD likewise invalidates everything after the divergence point. This is the problem the linker solves.

### 5.2 Single-module insertion (E1, E2, E8)

*Joint* = full prefill reference; *naive* = linker insertion after the prefix. E1 asks N=20 memory-recall questions over a 47-token prefix; E2 asks N=25 (20 memory + 5 prefix-recall) over an adversarial ~1k-token prefix — a realistic assistant conversation containing same-type distractors (other URLs, other schedules, other ticket ids) for the memory facts. Cells show joint / naive; the control column lists the no-memory score per model (Qwen · Llama · Coder).

| Experiment | Qwen3-4B | Llama-3.2-3B | Coder-7B | no-memory control |
|---|---|---|---|---|
| E1: short prefix (N=20) | 17 / **18** | 20 / **19** | 20 / **20** | 5 · 1 · 4 |
| E2: adversarial 1k prefix (N=25) | 23 / **23** | 23 / **24** | 24 / **24** | 4 · 8 · 9 |

**Insertion is indistinguishable from joint prefill in all three models** (±1 = noise: failure sets are identical across conditions and shared with the reference; the 7B achieves perfect parity at 20/20 and 24/24). The theoretically expected cross-attention deficit — module K/V never attended the prefix — does not manifest at this scale, even under adversarial interference. Sink-dropping and 64-token warm-splice ablations neither help nor hurt (18/20 each on Qwen E1), consistent with there being no deficit to repair. Setup latency usually favors the linker (e.g., E2 Qwen: 2.2 s vs 3.4 s; E1 Coder: 0.6 s vs 1.9 s), with one observed inversion (E2 Coder: 5.1 s vs 3.9 s): the advantage is I/O-bound versus compute-bound and thus hardware- and size-dependent, growing with module length.

**Scale check (E8).** A deterministic synthetic memory of 5,114 tokens (40 microservices × 10 unique attributes + 20 incidents; generator and seed in `bateria6.py`) with an adversarial 1k-token prefix and **N=60** questions, on Qwen3-4B: joint 51/60, **linked 50/60**, no-memory control 0/60. Eight of the ~10 failures are shared between conditions and the rest are same-type noise (the most confusable attribute across 40 services). Two properties strengthen with scale: the task is no longer saturated (85 % joint), so a deficit had room to appear and did not; and the setup advantage widens exactly as the O(bytes)-vs-O(compute) argument predicts — **1.7 s linked vs 12.1 s prefill (7.1×)**, up from 3.1× at 1.4k tokens. Module weight becomes operationally relevant at this size (754 MB at f16; ~400/210 MB at q8_0/q4_0 per §6, quality-free on this workload).

**The 14B point.** The same battery on Qwen3-14B Q4_K_M (machine B, full GPU) reaches **exact parity at a higher absolute score: joint 57/60, linked 57/60**, no-memory 0/60 — the insertion deficit still fails to appear as model capacity grows, consistent with §5.3's observation that the composition deficit *shrinks* with capability. The setup advantage also grows with model size on identical hardware: **0.82 s linked vs 2.87 s prefill (3.5×)**, versus 1.7× for the 4B on the same GPU — prefill cost scales with model depth and width while restore remains a byte copy (838 MB module), adding a third axis (model size, alongside compute regime and memory length) to the §5.6 economics.

**Pooled significance.** Individual cells here are underpowered (N=10–60); no single cell reaches significance in either direction. The claim of no insertion penalty therefore rests on the *pool*. Aggregating every single-module battery (E1, E2, E8/E8-rep across the applicable models) into one paired test — **420 paired questions** — linked insertion scores 78.6 % against joint prefill's 79.3 %: **no statistically detectable difference (McNemar exact *p*=0.69)**, with a 95 % Newcombe confidence interval on the joint−linked deficit of **[−1.7, +3.1] pp**, inside a 10 pp non-inferiority margin (we fix that margin *post hoc*, having not pre-registered it — an honest caveat). Extending the pool to the harder long-context and two-hop batteries (E14, E17; 600 questions total) leaves the verdict unchanged (Δ +0.2 pp, *p*=1.0). This replaces the earlier "no measurable loss" wording with a bounded, testable statement; the analysis recomputes from the versioned per-question outputs (`experiments/stats_recall.py`).

### 5.3 Two-module composition and the splice-k repair (E3, E5)

Modules A (1.4k tok) and B (0.3k tok) compiled independently and linked in sequence; N=10 questions per module.

| Condition | Qwen3-4B | Llama-3.2-3B | Coder-7B |
|---|---|---|---|
| joint (all prefilled) | 18/20 | 20/20 | 20/20 |
| composed (A then B linked) | 16/20 | 16/20 | 18/20 |

The deficit (~10–20 %) replicates across all three models, shrinks with model capability (−2 at 7B vs −4 at 3B against a perfect 20/20 base), and — pooled across models (N=140 paired questions) — is statistically robust, unlike the single-module case: **McNemar exact *p*<0.001, 95 % CI on the deficit [+7.4, +20.5] pp**. It has a specific signature: **cross-module attribution confusion** — e.g., asked for module B's database, the composed condition answers with module A's (PostgreSQL instead of SQLite). Modules never attended each other; under interference, fact-to-scope binding weakens. Per-module scoring shows the deficit lives entirely in the *later-linked* module: module A recall never degrades.

**Boundary recomputation repairs it (E5).** We evaluated fresh-decoded scope separators, recomputing the first k tokens of module B in context (splicing the precompiled remainder via `drop_k` linking), and their combination. Scores are module-B recall (N=10):

| Condition | Qwen3-4B | Llama-3.2-3B | Coder-7B |
|---|---|---|---|
| joint (reference) | 10 | 10 | 10 |
| composed, naive | 8 | 6 | 8 |
| scope separators | 7 | 7 | 8 |
| splice k=32 (~11 %) | 7 | 8 | 7 |
| **splice k=96 (~33 %)** | 7 | **9** | **9** |
| separators + splice32 | 6 | 7 | 7 |

Recomputing ~⅓ of the inserted module moves module-B recall back toward the reference on the two models with a clear deficit (Llama: monotonic in k, 6→8→9; Coder: 8→9), at the cost of prefilling 96 tokens — negligible. Separators alone contribute little. On Qwen3-4B the composed deficit was already within noise and no fixup moves it. **These per-condition cells are N=10 and individually non-significant; pooled across models at N=20/condition, splice-k *reduces but does not fully close* the two-module deficit (repaired 81.7 % vs joint 95.0 %, still McNemar *p*=0.039).** The clean, powered parity result for the composition recipe is at workspace scale (§5.7: N=120, *p*=1.0), not at this micro-benchmark. This is an in-situ confirmation of CacheBlend's thesis, implemented with nothing but rollback and offset linking. **Production recipe**: naive linking for single modules; splice-k with k ≈ 33 % of the inserted module for multi-module composition — whose quality-preserving parity is established at scale in §5.7.

### 5.4 Lazy loading of linked modules (E4a, E4b)

Full "classloader" scenario: a ~1k-token agent system prompt is prefilled; a 2.1k-token *general* memory module (containing the reference `[[memoria-ancla]]`) is linked after it; questions about the secondary project trigger loading the 0.3k-token linked module mid-conversation. General-memory recall on this linked base is 8/10 (Qwen3-4B), 9/10 (Llama-3.2-3B), and 9/10 (Coder-7B). We test two protocols: **E4a** links the module *after* the question has already been decoded; **E4b** (*load-then-requestion*) rolls back the ~20-token question, links the module, and re-decodes the question after it (~0.5 s total). N=10 per condition:

| Condition | Qwen3-4B | Llama-3.2-3B | Coder-7B |
|---|---|---|---|
| joint (reference) | 10/10 | 10/10 | 10/10 |
| E4a: module inserted after the question | 3/10 | 7/10 | 5/10 |
| E4b: **load-then-requestion** | **8/10** | 6/10 | **9/10** |
| no-load control | 1/10 | 3/10 | 3/10 |

Question-before-evidence ordering degrades every model (worst on Qwen3-4B, which answers "this is not in the memory", anchored on the general module's delegation statement). Load-then-requestion restores natural reading order and recovers accuracy to near-reference (9/10 on the 7B). The residual gap to 10/10 matches the E3 attribution deficit. Note that in a production agentic loop the requestion step is unnecessary: a model that defers its answer until after a tool result — the standard tool-use pattern — generates *after* the link, so causal attention already sees the module (§6.5).

### 5.5 Hybrid-model linking: Qwen3.5 / Gated DeltaNet (E7)

We validate the affine-composition linker of §3 on Qwen3.5-2B (Q4_K_M), a hybrid with 18 of 24 layers using Gated DeltaNet recurrence (per-layer state: 16 heads × 128×128 f32, a *fixed* ~20 MB per sequence regardless of context length) and 6 full-attention M-RoPE layers — entirely in Python over unmodified release binaries (`hibrido2.py`).

**Mechanics.**

1. Whole-state save/restore (§5.1-style, attention + recurrent) already works for hybrids: restored continuations are token-identical, so *prefix-shaped* modules for Qwen3.5 need nothing new.
2. `llama_memory_seq_add` asserts `n_pos_per_embd()==1` and is therefore unavailable for M-RoPE models: we replace the on-device K-shift with a **software rebase** — a NEOX delta-rotation of the serialized K rows applied host-side (for text, all four M-RoPE sections carry the same position, so M-RoPE ≡ NEOX; the same equivalence llama.cpp's own shift graph relies on), plus patching cell positions in the blob metadata. Verified against direct compilation at the target position: max K error 7.7e-2 (f16 double-rounding, not an algebra error). The linker thus requires *no* runtime position support at all.
3. The `PARTIAL_ONLY` state flag gives read/write access to just the recurrent section of the hybrid state.

**Affine extraction without a custom graph pass.** `S_M` is the module's zero-state response (one normal compile). `T_M` is extracted with **one probe per recurrent layer**: initialize layer ℓ's state to the identity and all other layers to zero — shallower layers then behave exactly as in the compile pass, so layer ℓ sees its exact compile-time inputs, and since the per-layer map is exactly affine given its inputs, `T_M[ℓ] = probe[ℓ] − S_M[ℓ]` with no ε-truncation error. Probing a random state against the affine prediction gives 4–7e-3 relative error (an orientation mistake would give O(1)). Compile cost is ×(L_recr+1) — 1282 tokens → 2 s base + 21 s probes, offline. The per-layer artifact is *constant-size*: `T` (1 MB) + `S_M` (1 MB) per layer for the 2B.

**Results** (E1-style battery, ChatML harness; recurrent rollback is not supported so per-question isolation uses a `seq_cp` checkpoint sequence). Cells show memory recall + prefix recall:

| Scenario | joint | naive (S:=S_M) | affine (T_M·S_P+S_M) | no-mem |
|---|---|---|---|---|
| 1: 45-tok prefix + 1282-tok module (20q mem + 2q prefix) | 20/20 + 2/2 | **20/20 + 2/2** | **20/20 + 2/2** | 4/20 + 1/2 |
| 2: ~1.1k-tok prefix + 294-tok module (10q mem + 6q prefix) | 10/10 + 5/6 | **10/10 + 6/6** | **10/10 + 6/6** | 1/10 + 6/6 |

Non-prefix insertion into a hybrid reaches **full recall parity with joint prefill** in both scenarios, under both recurrent-state policies. The result replicates exactly on Qwen3.5-4B (32 layers, 24 recurrent): 20/20+2/2 and 10/10+6/6 across joint/naive/affine, and an ablation that compiles the module directly at the target position (no software rebase) also scores 20/20+2/2 under both policies — attributing zero behavioral cost to the software rebase (max f16 K error 1.9e-1) and to the affine term alike. A second replication at 9B (Q8_0 weights) appears in §5.6.

State-space diagnostics separate the policies where the battery does not: relative L2 distance from the joint state is 0.379–0.410 (naive) vs 0.266–0.289 (affine) with the short module, ~0.20 vs ~0.19 with the long one — the affine term recovers real prefix-state information, and matters more the shorter the module (GDN's contractive gating decays `T_M·S_P` over long modules, making naive converge to affine). Link cost: 95–680 ms naive, 0.65–2.0 s affine.

We actively searched for a regime where the policies separate *behaviorally* and did not find one: placing eight session facts immediately before the link point (maximal recency, hence maximal expected reliance on recurrent state) still yields 8/8 recall under naive on both module lengths — the hybrid's full-attention layers (1 in 4) fully cover what naive discards from the prefix's recurrent state. The one recall failure observed is a prefix↔module *attribution* confusion shared by both policies (the §5.3 deficit class, a splice-k candidate), not a state-policy effect. **Naive state replacement is therefore the production recommendation for GDN hybrids** (cheaper, no `T` matrices in the module); the affine machinery is validated numerically and reserved for *purely recurrent* models (Mamba/RWKV without full-attention layers), where it is the only mechanism that can preserve the prefix at all — a natural next testbed. This closes the hybrid frontier of §3: the compatible set now includes GDN hybrids.

**A methodological trap worth reporting.** Our first 4B run showed an apparent degradation (naive 16/20, affine 11/20 — *below* naive) that was entirely a harness artifact: every failed answer was *empty*, because the thinking-mode model spent the whole generation budget inside an unclosed `<think>` block. Pre-filling the assistant turn with an empty think block (`<think>\n\n</think>`) restored full parity. Linked-state conditions can make a reasoning model *think longer*, which under a token cap masquerades as a quality deficit — batteries over thinking-capable models must pin the thinking mode or budget for it.

### 5.6 Cross-hardware, cross-runtime replication — and where precompilation pays (E8-rep, E9, E10, E12)

We replicated the headline experiments on machine B (Ubuntu 24.04, RTX 4070 Ti SUPER 16 GB, CUDA; same llama.cpp tag b10068 built from source, byte-identical GGUF files).

**Recall is platform-invariant.** The full E8 battery (5.1k-token memory, N=60) reproduces *exactly*: joint 51/60, linked-naive 50/60, no-memory 0/60 — identical scores across OS (Windows→Linux), backend (Vulkan→CUDA) and GPU vendor. The hybrid battery (E7) likewise replicates on Qwen3.5-9B at Q8_0 weights with full parity in both scenarios (20/20, 10/10 for joint, naive and affine alike), extending the GDN linker to a third model size and a second weight quantization. This is expected — serialization is backend-agnostic raw bytes (§5.1) — but it is worth stating as a measured fact: a `.kmd` module is a *portable* artifact, and link quality is a property of the method, not of the machine.

**Modules are machine-portable artifacts.** A GDN hybrid module compiled on machine A (Windows, probe extraction over Vulkan-built binaries) links on machine B (Linux/CUDA) in 59 ms with exact multi-fact recall — closing the loop on §5.1's backend-agnosticity for the hybrid case. Portability is *behavioral*, not bitwise: Vulkan and CUDA kernels do not produce identical floats, so cross-compiled modules differ in low-order bits; what is invariant is recall quality and, within a machine, token-identical restored generation.

**Scale check at 8–15k tokens (E10).** Three Wikipedia-bulk memories with injected synthetic facts (8.0k/10.2k/15.2k tokens, `[[linked]]`), compiled to q4_0 `.kmd` (1.39 GB vs 4.93 GB f16, 3.55× smaller) and linked from *disk* (read inside the setup timer, q4 cache both conditions): joint 110/120 vs linked **116/120** (no-memory 0/120), setup 2.0–3.0× faster even on the fast GPU. Notably, linked *outperforms* joint on the two larger memories (46 vs 42 of 48; 35 vs 33 of 36) — a directionally consistent hint that isolated compilation (memory tokens attending only to themselves, then rebased) protects recall against attention dilution by the preceding context. A replication on Coder-7B with f16 modules lands at parity (118 vs 117 of 120, both at ceiling), so we treat linked>joint as a single-model observation, not a general effect — but even parity raises the stakes: a precompiled module is never a *worse* memory in our data.

**The economic advantage is a function of compute, not a constant.** The same 5.1k-token module that gives a 7.1× setup advantage on an integrated-class GPU (Arc 140V, Vulkan) gives 1.7× on the RTX 4070 Ti SUPER; cold-restore TTFT gains range from 8.4× (CPU) and 4.3× (Arc, Vulkan) to 2.4× (vLLM on the RTX, E9). Restoring is O(bytes) while prefill is O(compute), so the ratio tracks how expensive compute is relative to memory bandwidth. This inverts the usual deployment logic: **precompiled memory is most valuable exactly where LLM agents are currently least practical — consumer and edge hardware**. On a machine that prefills a 5k-token agent memory in 12 s, the module makes persistent-memory agents *interactive* (1.7 s); on a datacenter GPU the same module is a modest latency win that instead pays as aggregate throughput and energy.

**Compute-regime sweep (E12).** We measured the regime dependence directly by sweeping GPU offload on a single machine: the 15.2k-token corpus memory on Qwen2.5-Coder-7B, f16 module (870 MB) read cold from disk inside the timer (page cache evicted before each read), full prefill vs restore, repeated five times per regime.

| Regime | Prefill | Restore (cold NVMe) | Ratio |
|---|---|---|---|
| CPU-only (20 cores) | 18.9 s (804 t/s) | 0.69 s | **27.6×** |
| Partial offload (12/28 layers) | 13.7 s | 0.72 s | 19.0× |
| Full GPU | 5.5 s (2781 t/s) | 0.78 s | 7.0× |

Values are medians of N=5 repeated runs; prefill spread is <3 % and cold-restore spread <2 % (min–max whiskers in Fig. 2).

![Figure 2: E12 compute-regime sweep (N=5 runs; bars are medians, whiskers min–max). The same 15.2k-token module on the same machine: prefill cost scales with available compute while restore cost is a flat, relocation-bound byte copy, so the precompilation advantage (7.0–27.6×) peaks exactly where compute is scarce. Restore is read cold from NVMe (page cache evicted each run) and included in the timer; its <2 % spread shows the cost is dominated by state relocation into device memory, not disk I/O.](figs/fig-regimes.png)

Recall is 6/6 in every cell and every one of the five runs. Restore time is essentially flat across regimes (rising only slightly toward full GPU as more state is uploaded to the device) — it is a relocation-bound byte copy, not a compute — while prefill cost tracks available compute; the ratio therefore *grows* exactly where compute is scarce. Nor does the flat restore depend on the module being cached in RAM: evicting the `.kmd` from the OS page cache before each timed restore (`posix_fadvise(DONTNEED)`, eviction verified by the file read slowing to NVMe speed) adds only ~0.1 s — **0.78 s cold vs 0.65 s warm at full GPU** (cold 0.69/0.72/0.78 s across the three regimes, N=5, <2 % spread) — because the upload and scatter of state into device memory dominates the cost, not the disk read. On NVMe-class storage, modules can be served directly from cold disk with no warm-up; slower media (HDD, network) would reintroduce an I/O term. One honest caveat: a 20-core desktop CPU still prefills a 7B at ~800 t/s, so "minutes vs milliseconds" claims require weaker CPUs (machine A's laptop-class CPU prefills at an order of magnitude fewer t/s), larger models, or longer memories; the scaling direction, however, is unambiguous and multiplicative in all three factors. For the home-lab agent with a large persistent memory — the setting that motivated this work — precompilation is not an optimization but the difference between feasible and not.

**The 51.8k-token point (E14).** Scaling the E8 generator ~10× (440 microservices across 11 regions plus 120 incidents, 51,790 tokens; seeded generator in `e14.py`) probes the length regime §7 previously listed as untested, on Qwen3-4B with the module read from disk inside the setup timer. Two dtype arms, each using the same cache type in *every* condition: q8_0 — joint 31/60, linked **34/60**, no-memory 0/60, setup 2.62 s vs 15.89 s (**6.1×**, up from 1.7× at 5.1k tokens on this GPU); f16 — joint 29/60, linked **32/60**, setup 4.79 s vs 13.85 s (the larger 7.6 GB module doubles link I/O relative to q8_0's 4.1 GB: at multi-GB module sizes the KV dtype directly buys link speed). Three observations. First, the linked module again never scores below joint prefill — ~90 % of failures are shared between conditions, the E8 signature of a task-level (not insertion-level) limit. Second, the absolute drop (from 85 % at 5.1k to ~52–57 % here, failures concentrated on the most confusable attribute across 440 services) afflicts joint prefill equally: it is an interference/capacity property of the model at this scale, and the f16 arm matching q8_0 rules out a quantized-cache cliff (extending §6.2's tolerance map for Qwen3-4B to 50k) — an ablation that was mandatory before attributing the cause, since §6.2's cliffs produce exactly this plausible-but-wrong failure mode. The parity claim at 50k is therefore "linking adds no cost", not "recall stays high" — though §6.7 shows the interference itself is removable: reading the same document page by page under a 4k budget recovers 49/60. Third, an ABI consequence: at this length the FA-off configuration used elsewhere in the paper stops being viable on a 16 GB GPU — without flash attention the materialized KQ scratch alone overflows VRAM (measured), and llama.cpp rejects quantized V caches without FA — so long-memory modules will in practice use the FA-on ABI (§6.1).

### 5.7 Multi-module workspace at 33k tokens and lazy loading at scale (E11)

The composition recipe of §5.3 (first module linked whole; each subsequent module spliced with k = 33 % recomputed) applied to all three E10 memories at once, on Qwen3-4B (q4 cache, 40k context): a **33.4k-token workspace assembled from three precompiled modules** versus joint prefill of the same tokens. All 120 questions:

| Condition | Recall | Setup |
|---|---|---|
| joint (33.4k prefill) | 105/120 | 9.25 s |
| workspace (1 link + 2 splice-k) | **106/120** | 8.18 s (1.13×) |

Composition at 20× the scale of §5.3 is statistically indistinguishable from joint prefill (McNemar exact *p*=1.0, 95 % CI on the joint−workspace difference [−7.9, +6.2] pp over 120 paired questions) — the powered counterpart to the underpowered §5.3 micro-benchmark. The setup advantage nearly vanishes on this GPU — splice-k recomputes ~6.1k tokens and the linker reads 1.39 GB from disk — so at datacenter-class compute the workspace pays in *quality-preserving modularity*, not setup time; on consumer hardware the §5.6 regime arithmetic applies.

**Lazy loading at scale.** With only the 15.2k-token memory linked, six questions about the 10.2k-token memory score 0/6 (confirming inter-module isolation — no leakage); linking that module mid-session (splice-k, 424 MB from disk) takes **1.13 s**, after which the same six questions score **6/6**. The §5.4 classloader pattern survives a 35× increase in loaded-module size with sub-second-scale latency.

### 5.8 Speculative decoding over restored state: the MTP draft head (E13)

Current model families increasingly ship a **multi-token-prediction (MTP) draft head** — an extra transformer block used for self-speculative decoding (DeepSeek-V3-style). This creates a compatibility question this section answers: does restoring precompiled KV state silently *break* speculation on such models?

**The gap.** In llama.cpp, the MTP draft context shares KV cells with the target context (the upstream `[TAG_KV_CACHE_SHARE_CELLS]` mechanism), and for such shared-cell caches every state function is an unconditional no-op — saving a draft context produces an empty blob and restoring reports success while writing nothing. The consequence is invisible by design: after a restore, the target model answers correctly (it verifies every speculated token), but the draft head attends over uninitialized KV for all restored positions, so acceptance drops and generation slows. We close the gap with a **~120-line patch to the runtime library** (`llama-kv-cache.{h,cpp}`, included in the repository under `patches/`): serialize only the layers the draft cache *owns*, and on restore — target first, draft second — locate the shared cells by position instead of allocating. Non-shared caches are byte-identical to the original code path.

**E13.** Qwopus3.5-4B-Coder-MTP Q6_K (a GDN hybrid, 32 blocks + 1 MTP block — one model exercising both §5.5 and this section), 1,284-token memory, 43-token question, 120 greedy tokens, draft depth 2, each phase on a cold server process (machine B, CUDA):

| Condition | Prompt tokens processed | Draft acceptance | Gen. speed |
|---|---|---|---|
| Full prefill (baseline) | 1,327 | 0.690 (69/100) | 203.7 t/s |
| Restore target + draft (patched) | **43** | **0.722 (70/97)** | 207.2 t/s |
| Restore, draft blob ablated (control) | 43 | 0.587 (64/109) | 185.2 t/s |

Acceptance over restored state matches the full-prefill baseline; ablating *only* the draft-head blob (5.0 MB, vs. 90.4 MB of target state) degrades acceptance and costs 9 % generation throughput while answers remain correct in all conditions — a causal demonstration that the serialized draft state is what preserves speculation. Two full reruns reproduce every acceptance ratio and token count exactly (greedy decoding); wall-clock throughput varies within ~1 % run-to-run (e.g. 207.1 vs 207.2 t/s restored), so the t/s figures above are from one run. The draft blob is thus a **correctness-neutral optional payload**: a runtime that cannot ingest it loses only speed, never answers — the right degradation semantics for a portable module section.

**Two methodological traps worth reporting.** Both initially masked the result behind a silent full re-prefill of the restored state. First, saving state one generated token past the memory boundary makes the follow-up request need a one-token rollback — impossible for the hybrid's recurrent state, so the server silently reprocesses everything. Second, BPE merging across the memory/question boundary breaks the token-prefix match even when the *text* is a prefix; the fix is cutting prompts at the exact common token prefix (token-array requests). With both fixed, the server extends restored hybrid state without re-prefill (43 tokens processed instead of 1,327) — which also corrects a belief we held mid-investigation: hybrid slot restore is not broken, merely *fragile*, failing silently on any boundary mismatch.

### 5.9 Sliding-window attention: the linker inherits the model's visibility (E20)

The last untested attention family within reach was interleaved sliding-window attention (SWA). On Gemma 3 4B (34 layers, 5 local : 1 global, window 1024 tokens), the full pipeline — compile, serialize, restore, non-prefix link behind the adversarial ~1k prefix, RoPE rebase — runs **unmodified** on llama.cpp's iSWA cache. Recall parity with joint prefill is exact when the module is commensurate with the window (16/20 = 16/20 short-prefix; 15/25 = 15/25 under the adversarial prefix; no-memory floors 0/20 and 5/25). With the 4.6k-token E8 memory — 4.5× the window — recall collapses *symmetrically* (joint 16/60, linked 14/60, no-memory 0/60): only the global layers see the whole memory, and attribute binding across 40 services fails under prefill and linking alike. The linked module therefore inherits the model's own visibility semantics — whatever the window would not have seen under joint prefill is equally unseen when linked — so the operative rule for a memory manager is simple: modules at or below the SWA window come for free; larger ones return what the model itself returns at that distance. Two-module composition shows the familiar §5.3 attribution deficit (18 vs 14 of 20; the splice-k repair is untested here). Storage economics improve: SWA layers serialize only their window, so the module costs 47 KB/token versus ~136 for an equally-sized full-attention module of this class (215.8 MB for 4,593 tokens) — sublinear in document length for the local layers.

## 6. Module ABI and a runtime-agnostic format

### 6.1 Compatibility axes

Measured or derived from code, a module is compatible with a linking context only along five axes:

1. **Exact model weights** — the module is a function of them; any weight update invalidates it.
2. **Tokenizer** — the stored token ids must re-tokenize identically.
3. **KV-cache dtype** — the serialized tensor bytes must match the context's cache type (§6.2).
4. **V layout**: `v_trans = !flash_attn` — the same command line can yield incompatible layouts on different GPUs under `-fa auto`, so FA must be pinned and recorded. Length forces the choice: at ~50k tokens FA-off is not viable on a 16 GB GPU (§5.6, E14), so long-memory modules will in practice standardize on the FA-on layout (the `convert` operation of §6.3 bridges the two).
5. **Backend is *not* an axis** — proven empirically (§5.1, §5.6): the same blob restores across Vulkan/CUDA/CPU.

llama.cpp validates only architecture name and per-layer shapes; nothing binds a module to its source text or exact weights — a stale module loads *silently*. A production format must therefore carry `{hash(weights), hash(tokenizer), hash(source MD), n_tokens, tokens[], kv_dtype, layout flags, format version}`, making module identity content-addressed and staleness detectable — the moral equivalent of solving Java's classpath hell up front.

### 6.2 KV dtype: free compression until a silent cliff (E6)

`mdc` supports all nine llama.cpp cache types; module sizes scale as predicted (f16 → q8_0 0.53×; q4_0/iq4_nl 0.28×) and position rebasing over *quantized* K (dequant → Hadamard → RoPE → requant) works end-to-end. A full E1-style battery per dtype on Qwen3-4B (E6: joint/linked at f16, f16+FA, q8_0, q5_1, q4_0, iq4_nl) shows every cell in the 17–18/20 noise band: at this scale, KV quantization down to 4-bit costs nothing measurable, the linked module never scores below joint prefill at any dtype, and FA on/off has no quality effect — modules get 3.6× smaller for free.

This does **not** generalize unconditionally: the tolerance is model- and scale-dependent. Qwen3-4B holds q4_0 KV up to 15k tokens (E10), but Qwen2.5-Coder-7B degenerates completely with a q4_0 cache at ~9k tokens (0/8 recall, repetitive output) while f16 is unaffected (7/8) — and the failure applies to *any* q4 cache, joint prefill included, so it is a property of the model×dtype×length combination, not of linking. A dtype sweep on the same bench places the cliff precisely: q8_0 matches f16 (7/8 both), while q5_1 already fails (0/8) — and fails *silently*, producing plausible but wrong numbers rather than degenerate text.

**Resulting production policy: q8_0 as the default module dtype** (half of f16 for free), with sub-q8 dtypes admitted only after per-model, per-target-length validation. A production `mdc` should automate that validation before compiling module stores.

### 6.3 The `.kmd` container and the `mdc` CLI

We implement the content-addressed format as the `.kmd` v0 container and the `mdc` CLI (`src/kmd/mdc.py`, installable via `pip install .`): `compile` with make-style staleness semantics; `index` (compiles a memory index plus all its `[[linked]]` MDs, warning on broken references); `verify` (detects stale source and wrong model without loading weights); `link` (the §5.3 splice-k production recipe built in); and `convert` (the FA↔non-FA layout transposition of §3, byte-identical on round-trip).

Hybrid models are first-class: `compile` detects the architecture from GGUF metadata, runs the per-layer probes, and attaches the `T_M` matrices plus RoPE parameters (needed because e.g. Qwen3.5-2B rotates only 64 of its 256 head dimensions) under a `hybrid` header; `link --recr naive|affine` applies the software rebase and the chosen state policy.

Module size is *not* stable across models: bytes/token = layers × GQA-KV width × dtype size, and token count varies by tokenizer — measured 147.5 KB/token (Qwen3-4B), ~109 (Llama-3.2-3B), ~57 (Qwen2.5-Coder-7B, whose 4-head GQA makes its modules 2.6× smaller than the 4B's). Weight quantization does not affect module size at all; only the KV dtype does.

### 6.4 Toward a runtime-agnostic interchange format (E9)

Because the mathematical content (per-layer, per-token K/V plus positions) depends only on (model, tokenizer, text, RoPE policy) and not on the serving engine, a canonical layout-neutral interchange format with per-runtime loaders (scatter into llama.cpp cell rows or vLLM 16-token paged blocks) is feasible — ELF/ONNX-style. vLLM integration additionally requires extending the scheduler-connector contract, which is prefix-shaped today — in fact narrower: the stock disk connector (E9) keys storage by a hash of the *entire prompt*, so KV is not reused even across different questions over the same memory. E9 nevertheless replicates the Phase-A result (§5.1) inside vLLM v1 with it: restored-KV answers are token-identical to full prefill (5/5 recall both) at 2.4× TTFT on an RTX 4070 Ti SUPER — the precompile/restore contract holds in a second, architecturally unrelated runtime. The required engine surface is small: MiniPIC (Ordonez and Parnell, 2026) demonstrates PIC primitives in a vLLM fork with under 100 lines of core-engine changes, which strengthens the case that the missing piece is a contract, not a rewrite. The MTP result of §5.8 sharpens the contrast between engine designs: vLLM registers MTP/EAGLE draft layers as first-class groups in its unified KV-cache manager, addressable by layer name through the same connector interface — so the draft section of a module needs *no* engine change there (the draft KV's one-position offset, which vLLM handles via its "last-block drop" rule, is a convention the interchange format must record), whereas llama.cpp's shared-cell design required the §5.8 serialization fix. The thickness of a per-runtime loader is a property of the engine's API surface, not of the format.

### 6.5 Beyond memories: skills, plans, and tool catalogs

The module abstraction is not specific to memories. Most of what surrounds an agent's context window is static, versioned text that is re-prefilled ritually on every session: skill definitions (SKILL.md), project conventions (AGENTS.md), standing plans (PLAN.md), and — the heaviest in practice — *tool catalogs*. Tools are declared as JSON Schemas but reach the model as ordinary tokens in the system prompt; an MCP-connected session routinely carries tens of thousands of tokens of schema text before the first user word (we observed ~23k tokens of active tool schemas, plus >100k deferred, in a production coding-agent session).

These artifacts are *better* compilation candidates than memories along every axis of §6.1: they are identical across users and sessions (compile once, amortize arbitrarily widely), centrally versioned (content-addressed identity is the natural fit), and increasingly loaded *on demand* — a skill invoked mid-conversation, a deferred tool schema fetched when first needed. On-demand loading is precisely non-prefix insertion: the artifact arrives after an unpredictable conversation prefix, exactly the case prefix caching cannot serve (§5.1) and the linker exists for (§5.3–5.4). The lazy-loading protocol of §5.4/§5.7 maps one-to-one onto skill and tool loading, with one simplification available for free in an agentic loop: because new tokens attend causally to everything before them, a model that defers its answer until after the tool result — the standard tool-use pattern — needs no re-question step at all: link the module when the tool returns, then generate.

### 6.6 Persistence by reference

Linked modules break an implicit invariant of today's LLM applications: that the conversation transcript is self-contained — that replaying its text reconstructs the state. A module contributes KV state but no transcript text, so a session that linked one can only be faithfully resumed by a runtime able to re-link it: serialized conversations must store *references* — `{module_id, position}` pairs — rather than content, and the content-addressed identity of §6.1 is exactly what such a reference needs (staleness and wrong-model detection included).

Two further system consequences follow. Context accounting must charge modules against the window even though they never appear as text — a 500-word chat may hold a 33k-token workspace (§5.7). And unloading is not symmetric with loading: evicting a module from the middle of a sequence requires rebasing everything after it, so eviction belongs between turns, under a manager with pinning (see the memory-manager design note in the repository, `docs/ARCHITECTURE.md`).

Both eviction paths are validated on live conversations (`e15.py`/`e15b.py`, findings H34 in the lab notebook). For full-attention models, in-place eviction plus compaction (`seq_rm` of the module's cells, negative `seq_add` over the tail) costs **0.5–1 ms** — the lazy K-shift lands on the next decode — and is behaviorally neutral: over seven turns with six module switches, isolated-battery recall is identical with paging (22/42) and with everything resident (21/42), while the bounded working set halves peak cell usage (16k vs 34k). For GDN hybrids the recurrent accumulator cannot be truncated per-token (its "eviction" is undefined: the module's contribution cannot be factored out of the fold), so the manager uses **checkpoint-and-replay** — a sequence-level snapshot before each link; on eviction, restore and re-decode only the post-module turns — evicting in ~5 ms at O(tail), never O(module), cost (41/42 vs the no-eviction control's 39/42 on Qwen3.5-4B). In both cases the conversation *keeps what was said*: answer tokens generated while a module was resident retain its content in their own KV rows (the model still quotes its first exchange verbatim after six compactions), so eviction loses only what was never asked. Finally, provenance: the model now "knows" content whose text never crossed the conversation, so the tool stub that links a module should declare the module's identity, keeping that knowledge auditable at the UI level.

### 6.7 Context virtual memory

The eviction of §6.6 plus one further primitive — **sealing a range of the live conversation as an anonymous module** (`seq_cp` of the range to a scratch sequence, serialize, evict and compact) — completes the middle layer that §2 identified as vacant between MemGPT's text-level paging and PagedAttention's physical block paging: a virtual-memory hierarchy for the context itself, where page-out preserves exact state and page-in is the linker.

**A conversation larger than the window (E16).** With `n_ctx` deliberately set to 4,096 and a watermark at 3,000 cells, a scripted 14-report conversation totalling **5,528 tokens** — which would abort on context overflow without sealing — ran to completion with 7 segments sealed to host RAM (244 ms each, dominated by serializing 58 MB). Isolation is exact: questions about sealed segments score 0/21 without page-in. Paging the sealed segment back (142 ms) recovers **15/21 — equal to or better than the 14/21 of segments that never left**. What bounds the conversation is no longer the window but storage; the resident working set is bounded by the watermark.

**Paged reading of a 51.8k-token document under a 4k budget (E18).** The E14 memory (identical generator, seed, and 60-question sample) was split at semantic boundaries into 28 chunks of ~2k tokens and **compiled page by page: 51,790 tokens in 8.8 s with `n_ctx` never exceeding 4,096** — the VRAM wall that §5.6's full-context run had to engineer around cannot occur by construction, and compilation is embarrassingly parallel. Reading uses a deterministic page table (the chunk containing the service or incident named by the question; selection and paging are thus measured separately — a production selector, §8, adds its own error term) with a 109 ms mean page-in:

| Condition | Recall | n_ctx |
|---|---|---|
| Full-context prefill (E14 joint) | 31/60 | 57,344 |
| Full-context link (E14 naive) | 34/60 | 57,344 |
| **Paged, one chunk per question (E18)** | **49/60** | **4,096** |

Paging at this scale is not an accuracy trade-off *once the correct page is selected*: it *outscores* the full context by 15 points while using a 14× smaller window, because the model reads a 2k-token page containing the relevant facts instead of attending across 440 interfering entries (the capacity limit diagnosed in §5.6). The residual failures are all one question template answered in the wrong format — none are page misses. Two honest qualifiers: E18's page table is a deterministic *oracle* that already knows which chunk holds each answer, so 49/60 is the ceiling a real selector (§8) works toward, not the end-to-end number it would achieve; and this experiment prices only the page-in, not the full economics (per-chunk compilation, multi-GB module storage, and the break-even against selective *text* retrieval), which §8 leaves as future work. The result replicates across models: under the same 4k budget, Qwen2.5-Coder-7B scores 46/60 (44 ms page-ins — its 4-head GQA makes pages 2.6× lighter) and Qwen3-14B a **perfect 60/60** (full-context references at 51.8k exist only for the 4B; for the 14B, whose single-hop score at 5.1k was 57/60, paged reading of a 10× larger document is at ceiling).

**Residency is not addressability.** Across seven independent measurements in this paper — E10 (twice), E14 (both dtype arms), E15b, E16, and E18 — a freshly linked module in a small context never scored below, and usually scored above, the same content resident in a large one. We read this as: a large context window acts as an *addressing* mechanism, not an attention guarantee — recall degrades with resident interference even under joint prefill, and paging exact state into a small working set restores it. The honest boundaries: selection must be solved (E18's page table was deterministic), and tasks requiring cross-page synthesis — multi-hop across chunks, whole-document summarization — cannot be served from a single page and remain the adverse case for paging designs (within one module, two-hop questions cost joint prefill and linked modules equally: aggregate 73 vs 74 of 120 across three models, E17).

## 7. Limitations

Small cells (N=10–25 per condition; N=60 in E8/E14, N=120 in E10/E11), nine small-to-mid instruction-tuned models (2B–14B), two GPUs plus a CPU point, synthetic Spanish-language memories, greedy decoding, substring-based scoring, and paired significance testing (McNemar exact, with Newcombe 95 % CIs) reported only on *pooled* cells — per-cell Ns (10–60) are individually underpowered, so no single cell is significant in either direction, and the 10 pp non-inferiority margin of §5.2 was fixed post hoc rather than pre-registered. A single synthetic factual-recall battery underlies the recall claims; more corpora, task types (instruction-following, reasoning, cross-span attribution), and module orderings/positions remain untested. The E2/E8 "no deficit" result is an absence of evidence at ~1k-token prefixes and up-to-51.8k-token modules, not a proof; at 51.8k absolute recall drops for joint prefill and linked modules alike (§5.6), so parity there means "no added cost", not "high recall". Two-hop questions within one module cost both conditions equally (aggregate 73 joint vs 74 linked of 120 across three models, E17), but with per-model variance in both directions (Coder-7B −5, Qwen3-14B +6 for the linked module; N=40 per cell) and multi-hop *across* pages under the §6.7 paging regime remains untested. The §6.7 sealing result (E16) uses one model; the paged-reading result (E18) replicates on three (49, 46, and 60 of 60), but its page selector is deterministic — a production selector adds its own error term. Module storage is heavy (f16 ≈ 147 KB/token for a 4B GQA model; ~36,000× the source text) though cold storage is cheap and the q8_0 default halves it — and hybrid recurrent layers invert this economics (constant size per module). E7's naive/affine parity means the affine machinery, while numerically validated, has not yet been shown *behaviorally* necessary (one hybrid family at three sizes, ≤1.3k-token memories). M-RoPE and GDN hybrids joined the compatible set in §5.5, MTP draft heads in §5.8 (one model, one quantization, and — unlike every other result — requiring our library patch until it lands upstream), and interleaved SWA in §5.9 (one model; beyond-window memories degrade equally under joint prefill and linking, and the splice-k repair is untested there); MLA (DeepSeek-family latent attention) and multimodal caches remain outside it. Backend coverage is likewise two runtimes — llama.cpp, whose downstream wrappers (Ollama, LM Studio, llamafile) inherit the mechanism unchanged, and vLLM, where our result is a prefix-restore feasibility check (E9, §6.4) rather than a working non-prefix linker: importing a `.kmd`, non-prefix insertion, and running the linker inside vLLM all remain future work. Other major serving stacks also cache KV hierarchically (SGLang's HiCache, TensorRT-LLM's cache reuse) but, like vLLM's native connector, remain prefix-bound — they reuse state produced by the engine's own execution rather than accepting relocatable external modules — so extending the linker to them is future work, not an expected incompatibility.

## 8. Future work

Smarter boundary recomputation (token selection by KV deviation à la CacheBlend, or by importance propagation à la KEEP, instead of a fixed prefix-k); extending the affine linker of §5.5 to *purely recurrent* models (Mamba/RWKV), where no attention layers exist to compensate and the affine policy is the only prefix-preserving mechanism — the natural testbed its validated machinery still lacks; extending `mdc`/`.kmd` v0 (implemented, §6.3) with a layout-neutral cross-runtime profile and principled compression — generic lossless compression is *not* it: measured on real modules, zlib recovers only 7–13 % on f16 blobs (quantized ones are less compressible still) while capping restore at ~300 MB/s, far below NVMe; the productive levers are the KV dtype (q8_0 halves size behaviorally free, §6.2) and CacheGen-style lossy delta encoding; extending the `mtp` section of `.kmd` (implemented as format v1 — modules package the patched server's target and draft state files with content-addressed identity, validated by byte-identical round-trip against the §5.8 artifacts) with draft-KV relocation and an in-process compile path, and upstreaming the serialization fix; a vLLM connector plus scheduler-contract extension (§6.4); scaling studies beyond the 51.8k-token and 14B points measured in §5.2/§5.6, toward the regime where per-question recall itself becomes the bottleneck and paging designs apply; multi-hop question workloads; and multi-module workspaces with dozens of lazily-loaded memories beyond the three of §5.7.

## 9. Conclusion

Treating agent memories as precompiled, relocatable, composable KV modules is not a speculative architecture: on stock llama.cpp binaries, single-module insertion at arbitrary positions shows no statistically detectable recall difference from joint prefill at every scale we tested (1.4k–15.2k tokens; pooled McNemar *p*=0.69, 95 % CI on the deficit within ±3 pp), a three-module 33k-token workspace matches joint prefill (*p*=1.0), lazy mid-session loading works at a 1 s cost, and the one quality gap found — multi-module attribution — is reduced by recomputing a third of the inserted module at negligible cost, closing to joint-prefill parity at workspace scale. The setup advantage ranges from 7.0× on a discrete GPU to 27.6× on CPU for the same 15k-token memory, growing exactly where compute is scarce — consumer and edge hardware. The approach is not even confined to pure-attention transformers: the same linker extends to linear-attention hybrids, where a memory module becomes a constant-size affine operator on the recurrent state — arguably a more natural "compiled memory" than per-token KV, and a capability no serving-side caching system currently offers. What distinguishes this work from the concurrent PIC line is not the relocation trick but the *object model*: KV state promoted from cache entry to artifact — persistent, portable across machines and backends, content-addressed, staleness-checked — compiled where compute is cheap and linked where it is scarce. The missing system component in today's runtimes is the linker, and this work shows it can be built from primitives they already ship.
